# Subplan 11 — P1-7 & P1-11: UTF-8-safe chunking and path normalization

**Status: Implemented.** Extracted from `docs/plans/11-poc-hardening.md`
(PR 6 of the phasing table). The remaining phases stay in the active plan.

## Problem

**P1-7 — Chunker can split multi-byte UTF-8 runes.** `Chunker.Split` sliced
text at raw byte offsets (`sizeBytes`, `overlapBytes`, and the `findBoundary`
fallback to `maxEnd`). For non-ASCII text (accents, CJK), chunk boundaries and
overlap starts could land mid-rune, producing invalid UTF-8 that was stored,
embedded, and displayed. The search result preview (`cli/search.go`,
`text[:120]`) had the same class of bug.

**P1-11 — Paths are not normalized.** Documents were keyed by the path string
exactly as typed. `tbuk ingest docs/` then `tbuk delete ./docs/a.md` (or an
absolute path) missed, and the same file could be indexed twice under two
spellings.

## Fix (delivered)

- **P1-7 chunker:** added `snapRuneStart`, which walks a byte offset backwards
  to the nearest `utf8.RuneStart`. `Split` snaps both the boundary returned by
  `findBoundary` and the overlap start (`boundary - overlapBytes`) to a rune
  start. Separator-based boundaries were already rune-aligned (the separators
  are ASCII), so only the `maxEnd` fallback and overlap offsets needed it.
- **P1-7 preview:** `TruncatePreview(s, n)` truncates to at most `n` runes
  (appending `...`), replacing the byte-slice `text[:120]` in
  `printSearchResults`.
- **P1-11:** `NormalizePath` resolves a path to an absolute, cleaned form via
  `filepath.Abs` (which also applies `Clean`, covering `./`, `../`, and
  trailing-slash spellings). It is applied at the ingest/update/delete boundary
  (`newIngestCmd`, `RunUpdate`, `RunDelete`, and the delete confirmation
  lookup). `NormalizePath` is idempotent, so the exported `Run*` funcs
  normalize internally without harm.

**Migration note:** existing relative-path document rows won't match after this
change (acceptable for a POC — the app has not been used). Re-ingesting a file
re-keys it under its absolute path.

## Tests (written first, TDD)

- `chunking`: `TestChunker_multibyte_utf8_chunks_valid` — 360-rune CJK text
  (3 bytes/rune, no separators, forcing the `maxEnd` fallback) yields chunks
  that are all valid UTF-8 with rune-aligned `StartByte`/`EndByte` (proved the
  bug first: mid-rune slices / invalid UTF-8).
- `chunking`: `TestChunker_accented_utf8_chunks_valid` — accented Latin
  (2 bytes/rune) with separators stays valid UTF-8.
- `cli`: `TestTruncatePreview_multibyteStaysValid` — 200 accented runes
  truncate to 120 runes + `...` and remain valid UTF-8;
  `TestTruncatePreview_shortUnchanged` leaves short strings intact.
- `cli`: `TestNormalizePath_absoluteUnchanged`,
  `TestNormalizePath_relativeResolvedAndCleaned` — absolute paths pass through,
  relative/`./`/`../` spellings resolve and clean.
- `cli`: `TestRunDelete_normalizesRelativeToAbsolute` — a document created under
  its absolute path is deleted when addressed by its relative spelling.
- `cli`: `TestRunUpdate_storesAbsolutePath` — ingesting via a relative path
  keys the document by its absolute path.
