# Subplan 11 — P0-6: hybrid MinScore + FTS5 query sanitizing

**Status: Implemented.** Extracted from `docs/plans/11-poc-hardening.md`
(PR 5 of the phasing table). The remaining phases stay in the active plan.

## Problem

- `Hybrid` ran both legs with `MinScore: 0` and never filtered the fused
  results, so `tbuk search --min-score 0.3` (default mode: hybrid) silently
  ignored the flag.
- `Keyword` returned `nil, nil` on *any* query error (`//nolint:nilerr`) to
  paper over FTS5 syntax errors from special characters. This also hid genuine
  DB failures, and inside `Hybrid` it meant keyword results silently vanished.

## Fix (delivered)

- `Hybrid` applies `opts.MinScore` to the fused RRF scores before truncating to
  TopK. RRF scores are `Σ 1/(60 + rank_i)` sums, not cosine values, so a hybrid
  `--min-score` is on a different scale from vector search — documented in
  `docs/initial-context.md`.
- `Keyword` sanitizes user input via `sanitizeFTS5Query`: each
  whitespace-separated term becomes a double-quoted FTS5 phrase (embedded
  quotes doubled), neutralizing operators and special characters. An
  all-whitespace / empty query returns no results without hitting the DB. Real
  query errors (e.g. a closed DB) now propagate wrapped with context instead of
  being swallowed by `nilerr`.

## Tests (written first, TDD)

- `search`: `TestHybridSearch_respectsMinScore` — chunk matching both legs
  (two-leg RRF ≈ 0.0325) survives a `MinScore: 0.02` filter while single-leg
  matches (≈ 0.0164) are dropped (proved the bug first: `want 1, got 2`).
- `search`: `TestKeywordSearch_sanitizesSpecialChars` — operator/paren/quote
  inputs (`foo AND (`, `"unterminated`, `NOT OR`, empty) never error, and a
  special-char query (`JWT!`) still matches the underlying `JWT` term after
  sanitizing.
- `search`: `TestKeywordSearch_dbErrorPropagates` — a closed DB surfaces an
  error instead of being masked as an empty result set.
