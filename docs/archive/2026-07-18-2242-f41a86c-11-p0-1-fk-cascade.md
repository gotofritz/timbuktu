# Subplan 11 — P0-1: Foreign-key cascade across pooled connections

**Status: Implemented.** Extracted from `docs/plans/11-poc-hardening.md`
(PR 1 of the phasing table). The remaining phases stay in the active plan.

## Problem

`storage.Open` ran `PRAGMA foreign_keys=ON` (and `journal_mode=WAL`) via
`db.Exec`. Both pragmas are **per-connection** in SQLite, but `database/sql`
maintains a pool: the pragma applied only to the one connection that executed
it. Every other pooled connection had `foreign_keys = 0`.

Consequences:

- `tbuk delete` could silently leave orphan chunks and metadata.
- Orphan chunks kept their `chunks_fts` rows (the FTS delete trigger fires on
  chunk deletion, which never happened), so deleted documents kept appearing
  in keyword/hybrid search.

## Fix (delivered)

- Apply pragmas in the DSN so every pooled connection inherits them:
  `file:<path>?_pragma=journal_mode(WAL)&_pragma=foreign_keys(1)&_pragma=busy_timeout(5000)`.
- Pin `:memory:` to a single connection via `db.SetMaxOpenConns(1)` (a second
  pooled connection to `:memory:` opens a *different* empty database).
- Add **migration 002** to repair already-damaged databases:

  ```sql
  DELETE FROM chunks   WHERE document_id NOT IN (SELECT id FROM documents);
  DELETE FROM metadata WHERE document_id NOT IN (SELECT id FROM documents);
  INSERT INTO chunks_fts(chunks_fts) VALUES('rebuild');
  ```

## Tests (written first)

- `TestForeignKeyCascade_AcrossPooledConnections` — file-backed DB with
  `SetMaxOpenConns(4)`; holds other pooled connections so the DELETE lands on a
  fresh connection, then asserts zero orphan chunks and zero stale FTS rows.
  Confirmed failing against the old `db.Exec` pragma code (2 orphans).
- `TestMigration002_RepairsOrphans` — seeds orphan chunks/metadata with FK
  disabled, re-runs migrations, asserts orphans removed and FTS rebuilt.
